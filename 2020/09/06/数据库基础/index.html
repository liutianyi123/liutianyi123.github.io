Chapter 1 Database System

为什么要选择数据库

一方面：当今世界数据无处不在(ubiquitous)具有普遍性(pervasive)

另一方面：数据库存储数据具有安全性(in a secure way)

数据和信息的关系

① 数据由原始的信息(raw facts)构成

② 信息来源于对原始数据(raw data)的处理

数据库简介

数据库是一种存储数据(stores data)的计算机结构(computer structure)具有共享性(Shared)和集成性(integrated)

数据库管理系统DBMS(Database management system)是一种管理数据库的结构(Manages the database structure)，并控制数据的访问(Controls access to data)还是一种程序的收集(Collection of programs)

数据库管理系统的作用和优势

数据库管理系统是用户于数据库之间的桥梁(intermediary)

① 实现数据的共享(sharing)

② 高效的管理数据(effective data management)

③ 向用户展现数据视图(view of data)

④ 改进共享、安全、集成、访问(access)、决策制定(decision-making)、生产力(productivity)等。

数据库种类

① 通过支持用户(user)数量划分

  • 一次支持一个用户：单用户数据库(Single-user database)

​ eg.桌面数据库(Desktop database):个人计算机上的单用户数据库

  • 一次支持多个用户：多用户数据库(Multiuser database)

​ eg.工作组数据库(Workgroup databases):支持少数用户或特定部门

​ 企业数据库(Enterprise database):支持多个部门的多个用户

② 通过位置(locaction)分类

  • 集中式数据库(Centralized database):数据位于单一站点(a single site)

  • 分布式数据库(Distributed database):数据分布在不同的站点(different sites)

  • 云数据库(Cloud database):使用云数据服务(cloud data services)创建和维护

③ 通过数据类型(data type)分类

  • 通用数据库(General-purpose database):包含在多个学科(multiple disciplines)中使用的各种数据

  • 特定学科数据库(Discipline-specific database):包含特定学科领域(specific subject areas)的数据

  • 运营数据库(Operational database):用于支持公司的日常运营(day-to-day operations)

​ eg.事务数据库(Transactional database)

  • 分析数据库( Analytical database):存储专门用于战术(tactical)或战略决策制定(trategic decision making)的历史数据(historical data)和业务指标(business metrics)

​ eg.数据仓库(Data warehouse):以用于决策支持的优化格式(a format optimized for decision support)存储数据

​ 联机分析处理(Online analytical processing (OLAP)):数据仓库检索(retrieving)、处理(processing)和建模(modeling)数据的工具

​ 商业智能(商业智能):捕获和处理(captures and processes)商业数据，以生成支持决策制定的信息

④ 通过结构化程度(Degree of structure)

  • 非结构化数据:以其原始状态(original state)存在

  • 结构化数据:源于格式化(formatting)

  • 半结构化数据:在一定程度上经过处理(process)

注意：NoSQL(不仅仅是SQL)主要处理非结构化数据

为什么数据库设计很重要

良好的数据库:便于数据管理(data management)，生成准确和有价值的信息

不良的数据库:导致错误和糟糕的决策(poor decision making)

文件系统数据处理的发展

手册的文件系统(Manual file systems):通过文件夹(file folders)
和文件柜(filing cabinets)系统管理的数据

计算机化的文件系统(Computerized file systems):基于计算机的系统( computerbased system)来跟踪数据(track data)并生成所需的报告

文件系统再现(File system redux):现代终端用户生产力工具,包括电子表格程序，如Excel

文件系统数据(File System Data)处理问题

开发时间长(Lengthy development times)

难以快速得到答案

复杂的系统管理(Complex system administration)

缺乏安全性(Lack of security)，数据共享(data sharing)受限

结构依赖

① 对文件的访问取决于它自身的结构(own structure)

② 所有文件系统程序都被修改以符合新的文件结构(conform to a new file structure)

结构独立

① 在不影响应用程序访问数据(the application’s ability to access the data)的情况下更改文件结构

② 结构独立性很重要，因为它大大减少了编程工作(programming effort )和程序维护成本(program maintenance costs)。

数据依赖

当数据存储特性(data storage characteristics)发生变化时，数据访问(Data access)也发生变化

数据独立

在不影响程序访问该数据的能力的情况下改变数据存储特性(Data storage characteristic)

总而言之：数据依赖的实际意义在于逻辑(logical)和物理格式(physical format)的区别

数据冗余(Data Redundancy)

描述：将相同的数据存储在不同的地方

  • 信息岛(Islands of information),即分散的(scattered)数据位置

  • 增加相同数据的不同版本的概率

结果：

① 数据安全性降低(Poor data security)

增加未经授权(unauthorized)查阅资料副本(a copy of the data)的机会

② 数据不一致(Data inconsistency)

当相同数据的不同版本和冲突版本出现在不同的地方时存在

③ 数据输入错误(Data-entry errors)

当在几个不同的文件中创建复杂的条目时，更有可能发生

④ 数据完整性问题(Data integrity problems)

保证数据的准确性和一致性->需要手动的努力

数据异常(Data Anomalies)

描述：没有完成所有必需的更改时进行开发

类别：

  • 更新异常

  • 插入异常

  • 删除异常

数据库系统(Database System)

描述：逻辑相关的数据存储在一个逻辑数据存储库(logical data repository)中

  • 分布在多个存储设施之间
  • DBMS消除了大多数文件系统的数据不一致(data inconsistency)、数据异常(data anomaly)、数据依赖(data dependence)和结构依赖(structural dependence)问题

数据库系统环境

数据库系统就是在数据库环境中定义(define)和规范(regulate)数据的收集(collection)、存储(storage)、管理(management)和使用(use of data)的组件的组织

DBMS功能

① 数据字典管理(Data dictionary management)

​ 数据字典:存储数据元素(data elements)及其关系的定义

② 数据存储管理(Data storage management)

​ 确保高效的性能

③ 数据转换和演示(Data transformation and presentation)

​ 数据的格式符合(conform to)逻辑预期(logical expectations)

④ 安全管理(Security management)

​ 加强用户安全和数据隐私(data privacy)

⑤ 多用户访问控制(Multiuser access control)

​ 确保多个用户可以同时访问数据库而不影响其完整性

⑥ 备份和恢复管理(Backup and recovery management)

​ 能够在失败后恢复数据库

⑦ 数据完整性管理(Data integrity management)

​ 最小化冗余，最大化一致性

⑧ 数据库访问语言和应用程序编程接口(Database access languages and application programming interfaces)

​ 查询语言:让用户指定必须做什么，而不必指定如何做

​ 结构化查询语言(SQL):事实上的查询语言和数据访问标准，被大多数DBMS供应商支持

⑨ 数据库通信接口(Database communication interfaces)

​ 通过多个、不同的网络环境接受终端用户请求

管理数据库系统:焦点转移

数据库系统的缺点：

① 成本高(Increased acquisition and operating costs)

需要精密的硬件和软件(sophisticated hardware and software)以及高度熟练的人员(highly skilled personnel)

② 管理复杂(Management complexity)

许多不同的技术接口(interface with many different technologies)

③ 保持时效性(Maintaining currency)

为了最大化数据库系统的效率

④ 依赖供应商(Vendor dependence)

需要在技术和人才培训方面进行投资

⑤ 更新与更换频繁(Frequent upgrade/replacement cycles)

总结

  • 数据由原始事实组成，通常存储在数据库中

  • 数据库设计定义了数据库结构，可以根据用户数量、位置以及数据使用和结构进行分类吗

  • 数据库从手工和计算机化的文件系统演变而来，因为文件系统数据管理有一些限制所以开发了DBMS

Chapter 2 Data models

数据建模和数据模型

数据建模(Data modeling)：为问题域(Problem domain)创建特定数据模型(specific data model)的过程(process)

  • 问题域(Problem domain)：在现实世界环境中明确定义的区域,有确定的范围(scope)和边界(boundaries)
  • 模型(Model)：对现实世界中更复杂的对象(complex object)或事件的抽象(abstraction)
  • 数据模型(Data model)：复杂现实世界数据结构(data structures)的简单表示
  • 模型的目的(Aim of a model)：帮助您了解现实环境的复杂性(complexities)

数据模型的重要性

数据建模的重要性不可夸大(overstate)

  • 促进沟通(Facilitates communication)

  • 提供数据库的各种视图(views)

  • 为各种用户组织数据(Organizes data)

  • 为创建良好的数据库提供了一个抽象

数据模型基本构建块

实体(Entity)：将收集和存储数据的人物、地点、事物或事件

  • 属性(Attribute)：实体的特征

  • 关系(Relationship)：实体之间的关联(association)

    • One-to-many (1:M OR 1..*)

    • Many-to-many (M:N or ..)

    • One-to-one (1:1 OR 1..1)

  • 约束(Constraint:)：对数据的限制

    • 确保数据完整性

商业规则

对政策，程序或原则的简短，准确且明确的描述

发现业务规则

业务规则的来源:

  • 公司经理 (Company managers)
  • 政策制定者 (Policy makers)
  • 部门经理 (Department managers)
  • 书面文件 (Written documentation)
  • 直接采访最终用户(必须验证)(Direct interviews with end users)

识别和记录业务规则的原因:

  • 组织运作的整体观点
  • 规范公司的数据视图
  • 促进用户和设计人员之间的交流工具
  • 协助设计师

将业务规则转换为数据模型组件

业务规则为正确标识实体、属性、关系和约束奠定了基础

  • 名词(Nouns)转化为实体(entities)
  • 动词(Verbs)转化为实体之间的关系

关系是双向的(bidirectional)

  • 识别关系类型的问题,例如：

    B的多少个实例与A的一个实例相关？

    A的多少实例与B的一个实例相关？

命名约定(Conventions)

实体名称要求:

  • 描述业务环境(business environment)中的对象
  • 使用用户熟悉的术语(terminology)

属性名：必须描述属性所代表的数据

  • 最佳做法：包括实体前缀

正确命名：

  • 促进双方之间的沟通
  • 促进自我解释(self-documentation)

层次模型和网状模型(Hierarchical and Network Models)

层次模型(Hierarchical models):为管理复杂制造项目(complex manufacturing projects)的大量数据(large amounts of data)而开发

  • 就像登月计划一样
  • 由包含段(segments)的倒立树(upside-down tree)表示
    • 段相当于文件系统的记录类型(file system’s record type)
  • 描述一组一对多（1：M）关系
    • 父母可以有很多孩子，但是一个孩子只有一个父母

网状模型：创建以有效表示复杂的数据关系(complex data relationships)

  • 改进了数据库性能(performance)并强加了数据库标准
  • 允许一个记录有一个以上的父对象（尽管被视为1：M）

随着网状模型而出现的标准数据库概念仍然被现代数据模型所使用

  • 模式(Schema)和子模式(subschema)

    • 模式是数据库管理员(database administrator)所看到的整个数据库的概念组织(conceptual organization)。
    • 子模式定义了应用程序“可见”的数据库部分 这些程序实际上可以从数据库中的数据中产生所需的信息。

  • 数据操作语言(Data manipulation language (DML))

    • 定义可以管理数据的环境，并用于处理数据库中的数据。

  • 数据定义语言(Data definition language (DDL))

    • 使数据库管理员可以定义架构组件(schema components.)。

关系模型(The Relational Model)（重点）

  • 关系型模型是IBM的Edgar Frank Codd在1970年引入的。
    • 关系模型的基础是一个被称为关系的数学概念。
  • 您可以将关系（也称为表(table)）视为由相交的行和列组成的二维结构 。
    • 关系中的每一行(row)称为元组(tuple)。
    • 每列代(column)表一个属性(attribute)。
    • 关系模型还基于高级数学概念描述了一组精确的数据操作构造。
  • 产生了一个自动传输数据库(automatic transmission database)，取代了标准传输数据库(standard transmission databases)
  • 关系数据库管理系统(Relational database management system (RDBMS))
    • 执行由层次和网络DBMS系统提供的基本功能
    • 使关系数据模型更易于理解和实施
    • 向用户隐藏关系模型的复杂性
  • 关系数据库结构类似于文件，但它是纯粹的逻辑结构(logical structure)
    • 生完整的数据和结构独立性
  • 基于SQL的关系数据库应用程序
    • 最终用户界面(End-user interface)
      • 允许最终用户与数据进行交互
    • 存储在数据库中的表的集合
      • 每张表是独立的
      • 不同表中的行基于公共属性中的公共值关联
    • SQL引擎
      • 执行所有查询

实体关系模型(The Entity Relationship Model)

数据库结构中实体及其关系的图形表示(Graphical representation)

  • 实体关系图(Entity relationship diagram)(ERD):使用图形表示为数据库组件建模
  • 实体实例或实体出现：关系表中的行
  • 属性：描述特定特征
  • 连接性：用于标记关系类型的术语

面向对象的数据模型(The Object-Oriented Data Model)

  • 面向对象的数据模型（OODM）
    • 数据及其关系都包含在一个称为对象的单一结构中
    • 面向对象的数据库管理系统（OODBMS）：基于OODM
  • 对象:包含数据及其与在数据上执行的操作的关系
    • 自治结构(autonomous structures)的基本构建块
    • 现实世界实体的抽象
  • 属性：描述对象的属性
  • 类：具有共享结构和行为的类似对象的集合
  • 类层次(hierarchy)：类似于上下颠倒的树，其中每个类都只有一个 一位父母
  • 继承：对象继承其上方类的方法和属性
  • 统一建模语言(Unified Modeling Language)(UML)：描述图和符号集，以 图形化地对系统建模

扩展关系数据模型(Extended relational data model)((ERDM))

支持面向对象(OO)特性、基于类的可扩展数据类型和继承

可扩展标记语言(Extensible Markup Language (XML)):管理非结构化数据，以便有效地交换结构化、半结构化和非结构化数据

新兴数据模型(Emerging Data Models:)：大数据和NoSQL

大数据目标

  • 寻找新的更好的方法来管理大量的网络和传感器生成的数据(web and sensor-generated
    data
  • 以合理的价格提供高性能

大数据的特征

  • 体量大(数量)
  • 速度快(生成和处理的速度)
  • 多样性(不同的数据格式)
  • 准确性(数据质量)
  • 价值(数据为业务创造价值)

大数据的挑战

  • 不允许使用常规结构
  • 价格昂贵
  • OLAP工具被证明在处理非结构化数据时不一致

大数据新技术

Hadoop, MapReduce, NoSQL

NoSQL数据库

  • 不基于关系模型

  • 支持分布式数据库架构

  • 提供高可伸缩性、高可用性和容错

  • 支持大量的稀疏数据(数据和属性,但实际你数据实例的数量很低)

  • 优化性能，而不是事务一致性

  • 为数据存储和操作提供了广泛的保护伞

抽象层次

Conceptual Medium-high Global view of data Hardware and software

总结

  • 数据模型是对复杂的真实数据环境的抽象
  • 有许多类型的数据模型(例如，分层的、网络的、关系的、面向对象的、扩展的关系数据模型等)。
  • 数据建模需求是不同数据视图(全局视图与局部视图)和数据抽象级别的功能
Model Advantages Disadvantages
Hierarchical Model 数据共享,概念简单,数据完整,数据库安全由DBMS提供,高效地处理1:M关系 需要物理数据存储特性的知识,层次路径知识,结构的变化需要所有应用程序的变化,实现的局限性,没有数据定义,缺乏标准
Network Model 概念简单，处理更多的关系类型，数据访问灵活，数据完整性，符合标准 效率低，结构的改变需要改变所有的应用程序
Relational Model 较强独立性,概念简单,特别查询功能基于SQL,将终端用户与物理级别的细节隔离,提高实施和管理的简洁性 开销大，可能导致信息问题
Entity Relationship Model 概念简单，视觉表达使其成为有效的沟通工具，与占主导地位的关系模型集成 有限约束表示，有限关系表示，没有数据操作语言，信息内容会丢失
Object-Oriented Model 添加了语义内容，视觉表示包含语义内容，继承促进数据完整性 开发缓慢，导航系统复杂，学习难度大，事务处理速度慢
NoSQL 容错性大，成本低，支持大数据，提高了存储效率 复杂的编程，没有关系支持，没有事务完整性支持

Chapter 3 The Relational Database Model

关系数据库的结构

关系数据库由表(table)构成，每个表有唯一的名字

关系(relation)用来指代表，元祖(tuple)用来指代行，属性(attribute)用来指代列，用关系实例(relation instance)表示一组特定的行

属性的取值集合被称作域(domain)。如果域中的元素被看作是不可再分的单元，则称域是原子的(atomic)

空(null)值表示值未知或不存在

数据库模式

数据库模式(database schema)是数据库的逻辑设计

数据库实例(database instance)是给定时刻数据库中数据的一个快照

关系模式(relation schema)类似于程序设计语言中类型定义，比如一个名叫department的表中有 dept_name,building,budget 三列那么它的关系模式为：department（dept_name, building, budget）

码

超码(superkey)是一个或多个属性的集合，通过这些属性的组合可以使我们唯一标识一个元组。

候选码(candidate key)是最小超码。

主码(primary key)是被设计者选中的候选码。

辅键码(secondary key)是用于数据检索目的的属性或属性组合。

一个关系模式(r1)包含另一个关系模式(r2)的主码，那么这个属性在r1中被称作外码(foreign key), 关系r1称作外码依赖的参照关系(referencing relation), 关系r2叫做外码的被参照关系(referenced relation)。

参照完整性约束(referential integrity constraint)指的是在参照关系中任意元组在特定属性上的取值必然等于被参照关系中某个元组在特定属性上的取值。

模式图

一个含有主码和外码依赖的数据库模式可以用模式图(schema diagram)来表示。

每一个关系用一个矩形来表示，关系的名字写在矩形上方，矩形内列出各属性。主码用下划线标注
